This module introduces {TSFName} ({TSFShortName}) and walks through the deployed environment -- showing the customer what they would get and how the components work together.

== Part 1: What is {TSFName}?

=== Know

Red Hat ships {OCPName}, RHEL, and hundreds of other products every year. To do this securely and at scale, Red Hat built Konflux -- an internal secure software supply chain platform. *{TSFName} is that same platform, packaged for customer deployment.*

This is not a reference architecture or a prototype. This is production software Red Hat trusts to build its own products.

The problem it solves is real: software supply chain attacks have grown dramatically. Attackers no longer break into production systems directly -- they compromise the build pipeline, inject malicious code into dependencies, or tamper with artifacts in transit. Customers need to prove that every container image they ship:

* Was built from verified source code
* Was not tampered with after the build
* Contains only known, scanned components
* Can be traced to a specific pipeline run and commit

{TSFShortName} makes all of that automatic. No custom tooling, no long-lived signing keys, no manual SBOM assembly.

*Talking points:*

* "Red Hat uses this to build most of its software -- {OCPName}, RHEL, Ansible -- you can use the same system."
* "Every image that leaves this pipeline is signed, scanned, and traceable."
* "When a new CVE drops, you know within minutes which images are affected."

== Part 2: The Architecture in Action

=== Know

{TSFShortName} deploys nine integrated components as Helm charts. When a developer pushes code, this is what happens automatically:

. *GitLab/GitHub webhook* triggers Konflux
. *OpenShift Pipelines (Tekton)* builds the container image
. *{RHTASName} (RHTAS)* signs the image using Fulcio (certificate authority) and records the signature in Rekor (tamper-proof transparency log)
. *{RHTPAName} (RHTPA)* generates an SBOM and scans for vulnerabilities
. *Quay* stores the signed image
. *Conforma* validates the release against ProdSec policies -- signature, provenance, and vulnerability thresholds must all pass
. *Release pipeline* promotes the image to the target environment only after all checks pass

The customer gets SLSA (Supply-chain Levels for Software Artifacts) Level 3 provenance automatically -- the highest attainable level without hardware roots of trust. This proves:

* *Who* built the image (build system identity)
* *When* it was built (timestamp)
* *How* it was built (pipeline definition and steps)
* *What source* was used (git commit SHA)

*Why SLSA Level 3 matters:* it is a requirement for federal software supply chain compliance under Executive Order 14028, and increasingly expected by regulated industries.

*Separation of concerns:* ProdSec defines the release policies enforced by Conforma. Platform Engineering manages the standard pipeline templates in Konflux. Developers add a component URL and write application code -- they do not write or maintain pipelines, signing configuration, or release rules.
