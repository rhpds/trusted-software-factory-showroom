= Module 2: Build, Sign, and Ship with Konflux

This module demonstrates the full secure pipeline -- from onboarding a component to verifying a cryptographically signed image with SLSA Level 3 provenance.

== Part 1: Creating an Application in Konflux

=== Know

Konflux models software as *applications* containing one or more *components*. An application is a logical grouping -- for example, a microservices system with a frontend, backend, and database service would be one application with three components.

When a component is added, Konflux automatically:

* Sends a pull request to the source repository adding `.tekton/` pipeline definitions
* Configures webhooks for push and pull request events
* Sets up integration tests
* Wires the build output to the signing and release pipeline

The developer does not configure Tekton, Fulcio, Rekor, or Quay separately. They add a component URL and Konflux handles the rest.

*Talking points:*

* "From the developer's perspective, they click 'Add component' and paste a Git URL. Everything else is automated."
* "TechCorp's security team mandated signed images. With {TSFShortName}, developers don't even know signing is happening -- it's part of the pipeline."
* "TechCorp's Platform Engineering team doesn't write or maintain these pipelines. Red Hat maintains the standard templates centrally. When ProdSec updates a security requirement, it flows through automatically -- developers just add a component and focus on code."

=== Show

. Click the *Konflux* tab and navigate to *Namespaces → default-tenant*.

. Click *Create application*.

. Enter the application name:
+
----
sample-application
----

. Click *Create application*.
+
[NOTE]
====
*Presenter tip:* "An application in Konflux is not necessarily a single service -- it is the grouping that Konflux uses to manage build, test, release, and signing policies. TechCorp might have one application per microservice or it might group several microservices in an application."
====

== Part 2: Onboarding a Component from Git

=== Know

Adding a component is where the magic starts. Konflux inspects the repository, detects the build strategy (Dockerfile, Buildpacks, etc.), selects the appropriate pipeline template, and automatically submits a pull request with the `.tekton/` definitions.

The pipeline definitions live in the developer's repository -- version-controlled, auditable, and reviewable via standard code review processes. This is a key security property: pipeline definitions cannot be changed without going through the same review gate as application code.

*Why this matters for security:* if a pipeline definition could be changed outside version control, an attacker could silently modify the build steps -- and the resulting image would still appear to come from the correct source. Storing `.tekton/` in the repo closes that gap.

*What this means for teams:* Platform Engineering manages the standard pipeline templates maintained by Red Hat. ProdSec requirements are already encoded in those templates. Developers who use company-standard pipelines are ProdSec-compliant from the first commit -- without maintaining their own task definitions or customizations unless they specifically need to.

Konflux detects the Dockerfile and selects the appropriate pipeline.

Konflux also automatically configured the webhook in GitLab -- future pushes and pull requests will trigger the secure pipeline without any manual setup."
====

== Part 3: Configuring Release Settings

=== Know

Konflux separates *building* from *releasing*. A build produces a signed deliverable, in our example a container image, and attaches provenance. A release promotes that image to a target environment according to a release plan.

Release plans define:

* Which images are eligible for release
* What policy checks must pass (signature verification, vulnerability thresholds, SLSA level)
* Where the image goes (Quay repository, environment namespace)

This separation means security policies are enforced at release time without depending on any manual checks.

=== Show

. In the *Terminal* tab, download and run the release setup script:
+
[source,bash,role="execute"]
----
oc get configmap setup-release \
  -n konflux-cli \
  -o jsonpath='{.data.setup-release\.sh}' > setup-release.sh
chmod +x setup-release.sh
----

. Run for your application:
+
[source,bash,role="execute"]
----
./setup-release.sh -a sample-application
----
+
[NOTE]
====
*Presenter tip:* "In production, release plans target specific environments -- dev, staging, prod -- and include approval gates, policy checks, and deployment strategies. For TechCorp, the security team controls the release policy. Developers cannot bypass it."
====
. Optional: view the release settings:
.. Switch to the *OCP Console* tab. 
.. In the menu, select *Administration -> CustomResourceDefinitions*.
.. Search for `ReleasePlan`.
+
Two custom resource definitions display: `ReleasePlan` and `ReleasePlanAdmission`. These custom resources define release parameters.
.. Click `ReleasePlanAdmission`.
.. Select the *Instances* tab and then click the name of the only ReleasePlanAdmission resource that displays (such as `local-release`).
.. Select the **YAML** tab.
+
[NOTE]
====
*Presenter tip:* "This resource contains the release settings for the application. The settings determine the applications that are included in the release, destination where the release image is pushed, and the Conforma policy that the image must pass in order to be released. The `policy: default` setting refers to the default Conforma policy, which requires that the signature of the image match the Rekor record for the build".

== Part 4: Triggering the Secure Pipeline

=== Know

Merging the Konflux pull request starts the full secure pipeline. From this point, Konflux automatically executes the following steps:

. Clone the source from the exact commit SHA
. Build the deliverable, such as a container image in our example, in an ephemeral, isolated environment
. Sign the image with RHTAS (Fulcio issues a certificate, Rekor records the signature)
. Generate an SBOM with all packages and dependencies
. Scan for vulnerabilities with RHTPA
. Push the signed image to Quay
. Run integration tests
. Trigger the release pipeline if all checks pass

The entire process is recorded. Every step, every input, every output -- all signed and stored in Rekor. This is the provenance record.

*Talking points:*

* "TechCorp's security team can query Rekor at any time and prove exactly when this image was built, by which pipeline, from which commit."
* "If a developer tries to build the same image manually and sign it with a different key, the signature will not match the Rekor record. The Konflux deployment pipeline includes a Conforma policy, which prevents the deployment of this image."

== Part 6: Verifying the Signed Image

=== Know

Signing without verification is theater. {TSFShortName} makes verification fast and unambiguous -- any engineer can confirm that an image came from the secure pipeline, was not tampered with, and has the expected provenance.

The verification uses `cosign`, the open-source tool from the Sigstore project. Cosign checks:

* The cryptographic signature against the Fulcio certificate
* The Rekor transparency log entry
* The SLSA provenance attestation

If any of these checks fail, the image is not from a trusted build. Period.

*Talking points:*

* "This check can run in a deployment pipeline. If an image is not signed with the expected identity, the deployment is rejected automatically."
* "No one at TechCorp -- not even a cluster admin -- can produce a passing cosign verify without going through the Konflux pipeline."

=== Show

. In the Konflux UI, navigate to *Applications → sample-application → Releases*.

. Click on the release name and then click the *Release artifacts* tab.

. From the *URL* column of the table, copy the image reference (format: `<quay-url>/<org>/<repo>@sha256:...`).
+
====
[IMPORTANT]
Select the image reference as text and copy it. Do not use the *Copy link* menu item as the link is different from the reference.
====

. Switch to the *Terminal* tab.

. Download cosign from the cluster's RHTAS service:
+
[source,bash,role="execute"]
----
tas_host=$(oc get route -A -l "app.kubernetes.io/part-of=trusted-artifact-signer,app.kubernetes.io/component=client-server" -o jsonpath='{.items[0].spec.host}')
ARCH=$(uname -m | sed 's/x86_64/amd64/;s/aarch64/arm64/')
curl --fail --location --output cosign.gz --show-error --silent https://${tas_host}/clients/linux/cosign-${ARCH}.gz
gunzip cosign.gz && chmod +x cosign
----

. Show what is attached to the image:
+
[source,bash,role="execute"]
----
IMAGE="<image-reference>"
./cosign tree $IMAGE
----
+
Expected output shows attestations (SLSA provenance, SBOM) and signatures.

. Initialize cosign to use the cluster's TUF root:
+
[source,bash,role="execute"]
----
TUF_HOST=$(oc get route -A -l "app.kubernetes.io/part-of=trusted-artifact-signer,app.kubernetes.io/component=tuf" -o jsonpath='{.items[0].spec.host}')
curl -sSfL -o /tmp/root.json https://$TUF_HOST/root.json
./cosign initialize --mirror=https://$TUF_HOST --root=/tmp/root.json
----

. Verify the signature:
+
[source,bash,role="execute"]
----
./cosign verify $IMAGE \
  --certificate-identity-regexp '.*' \
  --certificate-oidc-issuer-regexp '.*'
----
+
[NOTE]
====
*Presenter tip:* "Verification passed. This proves the image came from the Konflux pipeline and was signed by a certificate issued by the cluster's own Fulcio CA. No one outside that pipeline can produce this signature."
====

. Verify the SLSA provenance attestation:
+
[source,bash,role="execute"]
----
REKOR_HOST=$(oc get route -A -l "app.kubernetes.io/part-of=trusted-artifact-signer,app.kubernetes.io/component=rekor-server" -o jsonpath='{.items[0].spec.host}')
./cosign verify-attestation $IMAGE \
 --certificate-identity-regexp '.*' \
 --certificate-oidc-issuer-regexp '.*' \
 --rekor-url="https://$REKOR_HOST" \
 --type=slsaprovenance | jq -r '.payload' data.json | base64 -d | jq | less
----

== Part 7: Software Bill of Materials and Vulnerability Analysis

=== Know

An SBOM is a complete inventory of every software component in a container image -- operating system packages, language runtimes, libraries, and their versions. {TSFShortName} generates SBOMs automatically for every build and stores them in RHTPA, which continuously cross-references them against vulnerability databases.

When a new CVE is published, RHTPA immediately identifies every image in the registry that contains the affected package -- without re-scanning. This is the difference between reactive patching (rebuild everything, hope you find it) and proactive supply chain management (know exactly what is affected before the incident).

*Talking points:*

* "TechCorp's security team can search for any CVE and immediately see which applications are affected. That used to take days of manual analysis."
* "RHTPA also tracks license information -- useful for open-source compliance and legal review."
* "This is what executive orders and frameworks like NIST SSDF are asking for. {TSFShortName} generates it automatically."

=== Show

== Part 8: Introducing a CVE

=== Know

The standard build pipeline provided in TSF includes comprehensive scanning for vulnerabilities. If the scan discovers a known high-severity vulnerability, the build fails and the release does not proceed, preventing potential security incidents.

=== Show

. Click the *GitLab* tab and open `tsf/sample-component-golang` project.
. In the file list panel, click *Code -> Web IDE* . A new browser window opens with a VSCode user interface. Allow any requested permissions.
. Open the `demo/cve-onboarding` directory and select both the `go.mod.cve` and `go.sum.cve` files. 
. Right-click and select *Copy*.
. Click a file in the main directory.
. Right-click and select *Paste*.
. Using the right click and the *Rename* command, rename the `go.mod` and `go.sum` files to `go.mod.safe` and `go.sum.safe`.
. Using the right click and the *Rename* command, rename the `go.mod.cve` and `go.sum.cve` files in the main directory to `go.mod` and `go.sum`.
. Click the source control icon and then click *Commit and push to 'main'*. This push automatically triggers a Konflux build.
. Switch to the *Konflux UI* tab and then navigate to your application → Components.
. Wait until the build is reported as completed.
. Click the *Activity* tab and observe that the most recent pipeline displays a status of *Failed*.
. Click the name of the pipeline to see the steps. Observe that the `sample-application-enterprise-contract` step has failed.
. Click the `sample-application-enterprise-contract` step and then click *See logs*.
. Scroll to the bottom of the log output and observe that the step has failed because vulnerabilities were found.
. Optional: reverse the changes:
.. On the *GitLab* tab, use the Web IDE to delete the `go.mod` and `go.sum` files and rename the `*.safe` version back to `go.mod` and `go.sum`; commit and push the changes to `main`.
.. On the *Konflux UI* tab, navigate to your application → Components.
and then wait until the build is reported as completed.

*Talking points:*

* "If TechCorp's build contains a known critical vulnerability, nothing is deployed - ensuring that no attack surface appears."
* "When the failed deployment is investigated, the logs list the vulnerability."

[NOTE]
====
*Closing point for the demo:* "Every image that leaves this pipeline has this data automatically attached. TechCorp's security team has full visibility into what is in every container running in production -- and when the next Log4Shell drops, they'll know within minutes which workloads are affected. That is what a secure software supply chain looks like in practice."
